Detailed Description

template<typename T>
class tcg::list< T >

Double linked list container class with indexed access to elements.

This class is the result of implementing a double linked list using a dynamic random access sequential container as the underlying pool for the list's nodes.

This is somewhat different than supplying a custom allocator to an existing list implementation, like std::list, since:

  • No memory model of the underlying container is actually required.
  • The underlying container provides explicit access to nodes using their associated insertion indexes, so std::list could not be used as-is.


The nodes pool (see list_base) is an internal lazy container that saves released nodes without actually deallocating them (of course, their content is properly destroyed). They are rather stored into an internal stack of released nodes, ready to be reacquired.

Features
  • Insertion indexes are automatically generated keys with the same access time of the underlying container (typically super-fast O(1)).
  • Stable indexes and iterators with respect to every operation.
  • Copying an indexed list preserves the original indexes.
  • Indexes are process independent - they can be saved on file and restored later (although you probably want to squeeze the list before saving).
Use cases
This class is surprisingly useful for cross-referencing. A graph data structure, for example, can be implemented using a pair of indexed lists for vertexes and edges.
Known issues
The lazy nature of this container class may require squeezing the list's content at strategic places in your code, as noted before. Beware that composite cross-referencing may be hard to track precisely during this particular operation.
Remarks
Current implementation explicitly uses an std::vector class as the underlying random access container. Future improvements will make the container class a template parameter.
See also
Template class tcg::Mesh for a notable use case.

Warning
tcg::list has a different copy behavior from other container classes, since it is intended to preserve the source's node indices in the process.

This means that no value-to-value assignment can take place, and thus operator='s behavior is equivalent to first clearing it, and then copy-constructing each value. 

    This typically leads to performance penalties with respect to other

container classes that implement value-to-value assignments, like std::vector in case its capacity is not exceeded.
